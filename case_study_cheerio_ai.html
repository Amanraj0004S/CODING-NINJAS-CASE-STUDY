01 · Context

The Real Conversion Problem

Most businesses don't struggle with acquiring users — they struggle with converting and retaining them at the right moment. Cheerio AI positions itself as a multi-channel engagement platform designed to automate communication across the customer lifecycle.

The platform tracks user actions across touchpoints, triggers automated workflows, and sends messages across channels — primarily WhatsApp and Email. It attempts to improve conversion and retention. Critically: it operates on existing demand, not demand creation.

The Core Tension

Automation alone does not guarantee outcomes. The effectiveness of any engagement system depends on how well it understands user intent, timing, and context — not just whether it can fire a trigger.

02 · Funnel View

Where Cheerio AI Lives in the Journey

Using an e-commerce example, the user journey looks like this:

01User visits
02Adds to cart
03 — ENTRYDrops off
High intent moment
04Receives follow-up
05Converts or ignores

Cheerio AI enters at Step 3 — the high-intent drop-off moment. This is both its greatest opportunity and its greatest risk. The user has already signalled intent. The system's job is to honour that signal with precision — not volume.

03 · Key Problems

Five Structural Failures

01
Delivery ≠ Engagement
The system optimises for message delivery rates — open rates, send volumes. It does not optimise for what actually matters: whether the user took the intended action. Vanity metrics masquerading as product success.
02
Lack of User-State Awareness
Messages do not adapt to real-time user actions. A user who already purchased still receives a cart-abandonment flow. A user who engaged with support still gets a generic promo. The system has no memory.
03
Weak Personalisation — Rule-Based, Not Behaviour-Driven
Personalisation is primarily rule-based: "If X happens, send Y." There is no behavioural intelligence layer that asks: what does this user's full action history tell us about their next likely move?
04
Linear Workflows
Push-based messaging assumes the user is a passive recipient. Real engagement is bidirectional. A system that cannot respond to a reply, adapt to a click, or pause when the user signals fatigue is not an engagement system — it is a broadcast tool.
05
Weak Trust Signals
Published case studies lack critical context — no timeframes, no baseline comparisons, no attribution methodology. Without these, it is impossible for a prospective customer to determine whether results are genuine or cherry-picked.
04 · Root Cause Analysis

Why These Problems Exist

The five surface problems share a single root: the system was architected as a message delivery platform, not an intent understanding platform. The four underlying causes:

1

Rule-based workflows instead of behavioural intelligence — The decision engine cannot infer intent; it can only pattern-match to pre-defined rules.

2

Lack of unified user-state persistence — Each workflow operates in isolation. There is no shared user model that accumulates meaning across sessions and channels.

3

Shallow integration with client systems — The platform does not pull enough context from the client's CRM, order management, or support systems to know what the user has already experienced.

4

Workflow inefficiencies — no real-time decisioning — Triggers fire on scheduled batches, not real-time signals. The moment a user abandons a cart and the moment they receive a message are decoupled.

05 · Proposed System

The Intent-Driven Engagement Architecture

The shift required is not incremental. It is architectural. From: Message Automation. To: Intent-Driven Engagement System.

Layer 1 — Retention Engine
Post-Purchase Intent Mapping
  • Updates user state immediately after every significant action (purchase, support contact, return, review)
  • Builds intent hypotheses based on the accumulated action history: "This user is likely to churn in the next 14 days based on engagement decay pattern"
  • Validates hypotheses via low-friction interaction — a single question, not a survey
  • Suggests contextual next steps that match the user's inferred goal, not the brand's broadcast schedule
  • Optimises channel (WhatsApp vs Email vs in-app) and timing based on individual engagement windows, not global send times
Layer 2 — Conversion Engine
Pre-Purchase Profile Assembly
  • Retrieves full user context before any message is sent — what have they viewed, what have they ignored, what have they clicked
  • Builds a dynamic user profile through natural interaction: a question in message one becomes a data point that shapes message two
  • Adapts tone and message flow in real time: a user who responds quickly gets shorter messages; a user who reads slowly gets longer, more detailed ones
  • Reduces decision friction by surfacing the one piece of information most likely to unlock conversion for that specific user — not a generic discount
06 · User Experience Shift

What the User Actually Feels

Before — Current State
  • Generic messages that feel mass-produced
  • Multi-channel notifications that feel like spam
  • No memory of previous interactions
  • Discounts that signal desperation, not value
  • Low trust — brand feels automated, not present
→
After — Proposed State
  • Contextual messages that feel individually written
  • Fewer interactions, but each one relevant
  • Full memory of user journey — conversations, not campaigns
  • Value articulation that matches the user's specific hesitation
  • Higher trust — brand feels attentive, not automated
